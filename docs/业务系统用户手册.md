# APS 业务系统使用手册

文档版本：1.0  
适用对象：系统管理员、生产计划员、计划审批人、只读查看人员

## 1. 登录系统

浏览器访问业务系统地址，例如：

```text
https://aps.example.com
```

开发环境默认地址：

```text
http://127.0.0.1:8080
```

首次启动默认账号：

```text
用户名：admin
密码：admin123
```

正式环境首次登录后必须修改默认密码。当前 Web 页面支持管理员在“用户与权限”中重置用户密码；用户也可以通过 `PUT /api/auth/password` 修改自己的密码。

登录有效期由 `SESSION_HOURS` 控制，默认 12 小时。令牌过期后需要重新登录。

## 4. 总览中心

总览中心包含：

- 主数据记录数量；
- 累计任务数量和当前运行数；
- 待审批与已发布版本；
- 排程任务成功率；
- 最近排程任务；
- 算法服务健康状态；
- 快捷操作入口。

如果算法服务显示 `DOWN`，不要继续批量创建任务。先通知运维检查算法服务和网络。

## 5. 用户与权限管理

仅管理员可以使用。

### 5.1 创建用户

1. 打开“用户与权限”；
2. 点击“新建用户”；
3. 输入唯一用户名；
4. 输入显示名称；
5. 选择角色；
6. 设置至少 8 位初始密码；
7. 保持“账号启用”；
8. 保存并通知用户首次登录修改密码。

用户名创建后不可修改。如果录入错误，应停用旧用户并创建新用户。

### 5.2 修改角色

1. 点击用户行的“编辑”；
2. 修改角色；
3. 密码留空表示不修改；
4. 保存。

用户下一次 API 请求会按数据库中的最新角色判断权限。

### 5.3 停用用户

取消“账号启用”并保存。系统不允许管理员停用自己当前正在使用的账号。

### 5.4 重置密码

编辑用户，在密码框输入至少 8 位新密码并保存。管理员应通过安全渠道传递临时密码。

## 5.5 业务单元部署配置

“业务单元配置”只允许管理员修改，其他角色只读。配置项包括部署工艺、系统显示名称、部署锁、修改原因、修改人和修改时间。

| 部署类型 | 页面标题示例 | 新建任务工艺 |
| --- | --- | --- |
| `MACHINING` | 机加智能排程 | 固定机加，不可修改 |
| `HEAT_TREATMENT` | 热表智能排程 | 固定热表，不可修改 |
| `ASSEMBLY` | 装配智能排程 | 固定装配，不可修改 |
| `DEBUG` | 调试智能排程 | 默认为空，必须选择 |

存在排队中或运行中任务时不能切换业务单元；存在已发布计划时必须勾选明确确认；部署锁定后修改必须确认解锁。每次修改都必须填写原因并写入审计日志。

## 6. 主数据中心

主数据是算法输入的来源。创建任务时，系统会把当时的全部主数据固化为任务快照，后续修改主数据不会改变历史任务快照。

### 6.1 数据类型

#### 生产订单

订单包含工序列表。核心字段：

| 字段 | 说明 |
|---|---|
| `order_id` | 唯一订单编号 |
| `order_business_type` | 业务归属：`MACHINING` / `HEAT_TREAT` / `SURFACE_TREAT` / `ASSEMBLY` |
| `product_id / product_name` | 产品信息 |
| `material` | 材料类别或材料族 |
| `material_grade` | 精确材料牌号；存在允许合批工序时必填 |
| `quantity` | 订单数量 |
| `priority` | 业务优先级，数值越大通常越紧急 |
| `due_date` | 交期 |
| `release_date` | 可投产日期 |
| `status` | 订单状态 |
| `processes` | 工序数组 |

合批使用订单级 `material_grade`，比较前统一去除首尾空格并转换为大写。任一订单未填写材料牌号或牌号不同，都不能合批。

机加、热表、装配页签按 `order_business_type` 管理订单。热处理和表面处理订单共同显示在“热表订单”，但仍保留各自业务类型。任务选择订单后保留完整工艺路线，不会删除跨业务转工工序。

工序核心字段：

| 字段 | 说明 |
|---|---|
| `process_id` | 全局唯一工序编号 |
| `sequence` | 工艺顺序 |
| `status` | 工序状态 |
| `resource_group_id` | 候选资源组 |
| `unit_duration_minutes` | 单位工时 |
| `process_quantity` | 工序数量 |
| `previous_process_ids` | 前置工序编号 |
| `locks` | 时间、设备、人员等锁定信息 |

#### 设备档案

核心字段：

- `machine_id`；
- `machine_name`；
- `machine_type`；
- `workshop_type`；
- `status`；
- `capabilities`；
- `unavailability`；
- `availability_overrides`。
- `virtual_resource`，缺省为 `false`。

热处理设备还应维护 `max_volume_l` 和 `cooling_methods`。工序字段 `cooling_constraint_enabled` 默认 `false`，关闭时不检查设备或合炉成员冷却方式；启用为 `true` 时必须同时填写 `cooling_method`，否则主数据校验不通过。

#### 人员档案

核心字段：

- `worker_id`；
- `worker_name`；
- `skill_level`；
- `skill_efficiency`；
- `workshop_type`；
- `status`；
- 请假和加班时间段。
- `virtual_resource`，缺省为 `false`。

#### 资源组

资源组把工序需求映射到设备和人员：

- `resource_group_id`；
- `resource_group_type`；
- `machines`；
- `workers`；
- `machine_worker_mapping`；
- 可选 `toolings`。
- `virtual_resource`，缺省为 `false`。

资源组中的设备和人员必须已经存在于对应主数据中。

工序是否虚拟以资源组为准。虚拟资源组只能引用虚拟设备和人员；跨业务转工工序必须绑定虚拟资源组。虚拟资源无限并行但工时有限，不检查真实资源冲突、日历、容量、映射和资质，也不计入设备、人员利用率。

外部单位确认虚拟转工时间后，在工序 `external_plan` 中维护来源业务类型、外部单号、确认开始/结束时间、确认人和确认时间。外部确认来源必须与虚拟资源组 `resource_group_type` 一致，且不能与人工时间锁同时存在。

#### 工厂日历

工厂日历分为“机加日历”“热表日历”“装配日历”，每条日历使用 `schedule_type` 标识所属工艺。创建排程任务时，系统只把当前工艺对应的日历固化到算法快照中。

日历使用 `weekly_shifts` 定义每周工作时间，键值 `0` 到 `6` 表示星期日到星期六。还可使用 `special_shifts` 和 `special_rules` 表示特殊日期。热表日历默认使用每天 `00:00` 到次日 `00:00` 的连续时段，并通过 `day_shift_start: "08:00"` 保持新建任务的默认排程基准时间为下一工作日白班开始时间。

系统不再根据日历名称或编号猜测工艺类型。完整业务快照必须通过 `machine_calendars` 数组提供三类日历各一份。

### 6.2 新建与编辑

当前主数据编辑器使用 JSON，适合完整保留算法字段：

1. 选择数据标签；
2. 点击“新建记录”或现有行“编辑 JSON”；
3. 按 JSON 格式编辑；
4. 确认编号字段与路径编号一致；
5. 保存。

保存后记录修订号 `revision` 自动增加，审计日志记录操作人和时间。

### 6.3 删除

删除设备、人员或资源组前先检查引用关系。当前删除接口不会级联修改其他主数据，删除被引用记录会导致快照校验失败。

机加、热表和装配日历属于必需配置，页面不提供删除操作；需要调整时直接编辑对应类型的现有日历。

### 6.4 完整快照操作

主数据页签同一行右上角提供三个全局按钮：

- “校验快照”：检查当前全部主数据引用；
- “导出快照”：导出完整业务主数据对象，其中 `machine_calendars` 包含三类日历；
- “导入快照”：导入同结构完整业务主数据对象。

完整快照不是数组，必须包含以下五个根字段：

```json
{
  "machine_calendar": {},
  "machine_calendars": [],
  "machine_profiles": [],
  "worker_profiles": [],
  "resource_group_profiles": [],
  "order_processes": []
}
```

单独的订单、设备、人员或资源组数组不能使用“导入快照”，应使用下方当前页签的“批量导入”。

### 6.5 批量导入

生产订单、设备档案、人员档案和资源组页签提供“批量导入”。批量文件只包含当前页签的数据，根格式必须是 JSON 数组：

```json
[
  {},
  {}
]
```

例如设备批量文件：

```json
[
  {
    "machine_id": "MC_01",
    "machine_name": "加工中心 01",
    "machine_type": "加工中心",
    "workshop_type": "MACHINING",
    "status": "ACTIVE",
    "capabilities": {},
    "unavailability": [],
    "availability_overrides": []
  }
]
```

导入行为：

- 数组中的每一项必须是 JSON 对象；
- 每条记录必须包含当前类型的编号字段；
- 同一批量文件内编号不能重复；
- 同类型同编号记录覆盖并增加修订号；
- 新编号记录新增；
- 文件中未包含的旧记录不会自动删除；
- 每种工艺类型只使用一条工厂日历，不提供批量导入按钮。

### 6.6 批量导出

在生产订单、设备档案、人员档案或资源组页签点击“批量导出”，系统只导出当前页签的全部记录，下载内容同样是：

```json
[
  {},
  {}
]
```

建议在以下操作前批量导出留档：

- 大批量导入；
- 主数据结构调整；
- 重要排程；
- 系统升级；
- 问题复现。

### 6.7 校验快照

点击“校验快照”。系统检查：

- 是否存在有效周日历；
- 资源组引用的设备、人员是否存在；
- 工序编号是否重复；
- 工序引用的资源组是否存在；
- 订单和工序是否缺少编号。

业务系统校验通过后，算法系统还会进行更完整的结构、状态、时间和工艺约束校验。

## 7. 创建排程任务

计划员或管理员可以创建任务。

### 7.1 操作步骤

1. 打开“排程任务”；
2. 点击“新建任务”；
3. 正式部署核对固定工艺；调试部署选择工艺类型；
4. 选择排程模式；
5. 选择派工规则；
6. 检查默认填入的下一个工作日白班开始时间，必要时调整排程基准时间；
7. 动态滚动模式检查保护窗口，默认 24 小时；
8. 检查七项 TOPSIS 权重，默认合计为 1；
9. 检查其他算法参数覆盖 JSON；
10. 局部模式填写局部调整 JSON；
11. 提交。

系统会：

1. 校验当前主数据引用；
2. 生成全局唯一 `task_id`；
3. 固化完整数据快照；
4. 把任务标记为 `QUEUED`；
5. 后台调用算法系统；
6. 根据结果更新为 `SUCCEEDED` 或 `FAILED`；
7. 成功后自动生成 `DRAFT` 计划版本。

### 7.2 工艺类型选择

正式部署由业务单元配置固定工艺，前端只读，后端也会强制覆盖请求值。调试部署默认不选择工艺，提交前必须明确选择机加、热表或装配。

一次任务只处理一种任务工艺，但订单一旦选中会保留其全部工序。热表任务同时选择 `HEAT_TREAT` 和 `SURFACE_TREAT` 订单，二者不会跨类型合炉或合槽。

### 7.3 模式选择

#### 静态全量

适合首次计划、周计划、月度计划或需要全面重算的场景。

#### 动态滚动

适合：

- 紧急插单；
- 设备故障；
- 人员请假；
- 物料延期；
- 报工与原计划偏差；
- 每日滚动更新。

动态模式会保护进行中、确认和窗口外任务。

#### 局部微调

适合指定工序换时间、换设备或换人员。真实工序可在资源组范围内换机、换人；虚拟工序只允许调整预计转工时间，不能选择设备或人员。

示例：

```json
[
  {
    "process_id": "WO_001_P20",
    "plan_start_time": "2026-07-14T08:00:00",
    "plan_end_time": "2026-07-14T12:00:00",
    "assigned_machine_id": "MC_01",
    "assigned_worker_id": "W_01",
    "local_adjustment_authorized": true
  }
]
```

### 7.4 TOPSIS 权重

新建任务默认权重：

```yaml
makespan: 0.15
tardiness_count: 0.20
total_tardiness: 0.15
machine_idle_rate: 0.20
machine_balance: 0.15
worker_balance: 0.05
wip_waiting: 0.10
```

页面实时显示权重合计，提交时要求七项权重之和为 1。

### 7.5 动态滚动保护窗口

选择“动态滚动排程”后显示保护窗口，默认 24 小时。该值写入：

```json
{
  "rolling_window": {
    "confirmed_protection_hours": 24
  }
}
```

算法会保护计划开工时间前后该窗口范围内的 `CONFIRMED` 工序。

### 7.6 其他算法参数覆盖

页面默认提供较小参数，适合演示和快速联调：

```json
{
  "nsga3": {"population_size": 8, "generations": 3},
  "tabu_search": {"enabled": false}
}
```

正式排程应根据规模和排程质量要求配置。不要盲目使用联调参数作为生产标准。

动态窗口示例：

```json
{
  "rolling_window": {"window_days": 7},
  "nsga3": {"population_size": 48, "generations": 30},
  "tabu_search": {"enabled": true}
}
```

## 8. 任务状态与处理

| 状态 | 含义 | 用户操作 |
|---|---|---|
| `QUEUED` | 已创建，等待后台执行 | 等待 |
| `RUNNING` | 正在调用算法 | 等待并观察算法状态 |
| `SUCCEEDED` | 算法成功 | 到计划版本查看 |
| `FAILED` | 数据、网络或算法失败 | 查看错误，修复后重试 |

点击“详情”可查看：

- 工艺类型和模式；
- 独立的“请求数据”区域，展示任务参数和固化快照；
- 独立的“结果数据”区域，展示算法状态、排程明细、KPI 和候选方案；
- 错误信息；
- 创建、开始和结束时间。

### 8.1 失败重试

只有 `FAILED` 任务可以重试。重试复用原 `task_id` 和原始快照，适合算法服务暂时不可用等情况。

如果失败原因是主数据错误，并且已经修改了主数据，应新建任务以生成新快照，不要重试旧快照。

## 9. 计划版本

算法成功后生成版本编号：

```text
PLAN-YYYYMMDD-NNNN
```

状态流转：

```text
DRAFT -> APPROVED -> PUBLISHED
   |                    |
   +-> REJECTED         +-> 后续同工艺版本发布后变为 SUPERSEDED
```

### 9.1 查看版本

打开计划后可查看：

- 工艺类型和来源任务；
- 标题区域显示本版本的综合得分，得分取算法结果排名第一的方案；
- 最大完工时间、总延期、设备利用率、按期交付率和等待时间 KPI；
- 订单工序、设备、工人三个甘特图页签，点击页签自由切换；
- 订单工序甘特图可按订单展开或收起内部工序；
- 工序计划明细；
- 设备、人员、计划开始和结束时间。

鼠标移入甘特图工序柱时，页面显示订单号、工序号、分配设备、分配工人、物料齐套时间、计划开始时间和计划完工时间。热处理工序存在合批时，还会显示合批号。

排程明细会显示工序当前状态。只有业务人员主动执行锁定操作时，工序才显示“人工锁定”标识；系统排程、滚动窗口保护、进行中保护等运行期约束不会写入 `locks`。无人工锁时数据固定为 `"locks": {}`。点击人工锁标识可以查看锁定设备、锁定工人、锁定开始和结束时间、操作人、锁定时间、锁定原因及当前资源分配。

在“当前生效排程”的订单工序甘特图中，管理员或计划员可以在工序柱上点击鼠标右键：

- 选择“人工锁定”或“修改人工锁”，可锁定设备、人员和计划时间，并必须填写锁定原因；
- 当前工序已有排程时，设备、人员、开始时间和结束时间自动带入当前排程值；
- 设备、人员和时间可按需保留，开始和结束时间必须同时填写，且至少选择一种锁定内容；
- 热表工序锁定设备时会校验冷却方式，设备和人员会校验资源组、启用状态及映射关系；
- 人工锁时间与其他人工锁使用同一设备或人员且发生重叠时，系统拒绝保存；同一合批内的工序除外；
- 人工锁和局部调整会调用算法系统的统一工艺约束服务，与全量排程使用完全相同的资源组、设备人员映射、人员资质、冷却方式、容量、工时和日历规则；
- 找不到设备/人员共同可用日历窗口时直接判定不可行，不会用简单的时间相加生成伪计划；
- 选择“解除人工锁”时必须填写解锁原因，解锁后立即恢复为 `"locks": {}`；
- 锁定、修改和解锁均写入审计日志；如果计划版本或人工锁已被其他用户更新，页面会要求刷新后重新操作，避免覆盖他人修改。

KPI 中算法返回的 `makespan`、`total_tardiness` 和 `wip_waiting` 以分钟计，页面自动换算为小时；设备利用率由 `1 - machine_idle_rate` 计算；按期交付率由订单总数和延期订单数计算。

### 9.2 审批检查清单

审批人至少检查：

- [ ] 工序数量完整；
- [ ] 交期风险可接受；
- [ ] 紧急订单优先级符合业务要求；
- [ ] 设备和人员安排合理；
- [ ] 维护、请假和加班已正确体现；
- [ ] 工序前后关系正确；
- [ ] 热处理合批符合工艺要求；
- [ ] 锁定和进行中工序未被非法移动；
- [ ] KPI 相比当前计划没有不可接受的恶化；
- [ ] 需要下发的生产范围明确。

### 9.3 审批通过

点击“审批通过”，填写审批意见。版本进入 `APPROVED`，版本结果中的工序由 `PENDING（待排程）` 转为 `SCHEDULED（已排程）`，但尚未回写订单，也不是当前生效计划。

### 9.4 驳回

点击“驳回”，必须清楚填写原因。版本结果中的工序保持 `PENDING（待排程）`。`REJECTED` 版本不能再次审批，应修复数据或参数后创建新任务。

### 9.5 发布

只有 `APPROVED` 版本可以发布。发布会：

1. 把同工艺原 `PUBLISHED` 版本改为 `SUPERSEDED`；
2. 当前版本改为 `PUBLISHED`；
3. 把结果回写对应订单工序；
4. 写入计划开始、结束、设备、人员和版本号；
5. 把版本结果及订单工序由 `SCHEDULED（已排程）` 转为 `CONFIRMED（已确认）`；
6. 写入计划硬锁；
7. 记录发布人、时间和审计日志。

发布属于重要业务操作。发布前应确认没有另一位审批人正在处理同工艺版本。

新版本的工序状态流转固定为：`PENDING（待排程） --审批通过--> SCHEDULED（已排程） --发布下发--> CONFIRMED（已确认）`。算法完成资源分配但版本尚未审批时仍属于 `PENDING`。系统不对历史版本执行状态兼容或迁移。

## 10. 版本对比

在“计划版本”点击“版本对比”：

1. 选择基准版本；
2. 选择对比版本；
3. 开始对比。

系统比较：

- 两个版本的综合得分、得分差值、变化率以及提升或下降方向；
- 新增或删除的工序；
- 计划开始和结束时间；
- 设备；
- 人员；
- 热处理批次；
- 前后 KPI。

建议在动态排程和局部微调后，始终与当前发布版本进行对比。

## 11. 审计日志

管理员和审批人可以查看。主要动作包括：

- 用户登录、创建和修改；
- 密码修改；
- 主数据保存、删除和导入；
- 任务创建、失败、成功和重试；
- 版本审批、驳回和发布；
- 演示数据初始化。

审计日志用于问题追踪，不应被当作完整安全审计平台。生产环境还应保留网关、操作系统和数据库备份日志。

## 12. 标准业务流程

### 12.1 首次静态排程

1. 管理员创建计划员和审批人；
2. 计划员导入正式主数据；
3. 导出一份快照备份；
4. 执行快照校验；
5. 创建小参数试算；
6. 核对结果数据完整性；
7. 使用正式参数创建静态排程；
8. 审批人检查 KPI 和甘特图；
9. 审批通过；
10. 发布并检查工序回写。

### 12.2 每日动态滚动

1. 更新订单、报工、设备、人员和物料状态；
2. 校验快照；
3. 选择动态模式和滚动基准时间；
4. 创建任务；
5. 与当前发布版本对比；
6. 重点检查保护区和进行中任务；
7. 审批并发布。

### 12.3 紧急插单

1. 新增紧急订单并设置高优先级和真实交期；
2. 确认物料就绪时间；
3. 创建动态任务；
4. 对比插单前后受影响订单；
5. 评估延期变化；
6. 审批发布，并通知受影响部门。

### 12.4 局部换机换人

1. 确认目标工序及新时间；
2. 确认设备、人员属于工序资源组；
3. 确认共同日历可用；
4. 输入局部调整 JSON；
5. 创建局部任务；
6. 检查受影响工序和下游工序；
7. 对比原发布版本；
8. 审批发布。

## 14. 常见问题

### 14.1 为什么修改主数据后旧任务结果没有变化

每个任务保存独立快照。旧任务和旧版本不会随主数据变化。需要创建新任务。

### 14.2 为什么重试仍然报同样的数据错误

重试使用原始快照。数据错误应修改主数据后新建任务。

### 14.3 为什么不能审批或发布

检查角色和版本状态。只有 `admin / approver` 可以审批发布，且只有 `DRAFT` 可审批、`APPROVED` 可发布。

### 14.4 为什么发布后旧版本变成 SUPERSEDED

同一种工艺只保留一个当前发布版本，新版本发布后旧版本自动转为历史版本。

### 14.5 为什么主数据删除后任务无法创建

其他记录仍引用被删除的设备、人员或资源组。执行快照校验并修复全部引用。

### 14.6 为什么算法显示正常但任务失败

健康检查只表示服务进程可访问。查看任务错误，可能是快照字段、日历、资源映射、状态或工艺约束问题。

### 14.7 为什么页面提示登录过期

登录时长已达到 `SESSION_HOURS`，或管理员修改了 `SESSION_SECRET`。重新登录即可。

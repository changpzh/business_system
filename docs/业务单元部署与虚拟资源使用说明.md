# 业务单元部署与虚拟资源使用说明

文档版本：2.0  
更新日期：2026-07-16  
数据口径：按最新模型直接运行，不自动迁移旧订单归属、旧锁字段或无 `schedule_type` 的日历。

## 1. 部署模式

业务系统部署后固定服务于一个业务单元。系统配置支持：

| 配置值 | 页面标题 | 可创建任务 |
| --- | --- | --- |
| `MACHINING` | 机加智能排程 | 机加 |
| `HEAT_TREATMENT` | 热表智能排程 | 热表 |
| `ASSEMBLY` | 装配智能排程 | 装配 |
| `DEBUG` | 调试智能排程 | 机加、热表、装配 |

正式部署的新建任务弹框显示固定工艺且不可修改，后端会强制使用部署工艺。调试部署的工艺类型默认为空，用户必须选择后才能提交。

## 2. 配置页面

“业务单元配置”页面包含：

- 部署工艺类型；
- 系统显示名称；
- 部署锁定状态；
- 修改原因；
- 配置修改人；
- 配置修改时间。

只有管理员可以修改。系统保存完整审计记录，并执行以下保护：

1. 存在 `QUEUED`、`RUNNING` 任务时禁止切换；
2. 存在已发布计划时必须明确确认；
3. 配置锁定后必须明确确认解锁；
4. 每次修改必须填写原因；
5. 正式环境建议修改完成后立即锁定。

系统左上角标题直接使用配置中的 `system_display_name`。

## 3. 订单业务类型

订单新增：

```json
{
  "order_business_type": "MACHINING"
}
```

允许值为：

```text
MACHINING
HEAT_TREAT
SURFACE_TREAT
ASSEMBLY
```

机加部署只显示机加订单；热表部署同时显示热处理和表面处理订单；装配部署只显示装配订单；调试部署显示全部订单。

任务快照按订单业务归属生成。订单进入任务后，全部工序都会进入算法，不再删除跨工艺转工工序。

热处理和表面处理订单都进入热表任务，但订单仍分别保留 `HEAT_TREAT`、`SURFACE_TREAT`。真实资源组也分别使用这两个 `resource_group_type`，两者不能相互合炉或合槽。

## 4. 虚拟资源维护

资源组、设备和人员支持：

```json
{
  "virtual_resource": true
}
```

虚拟资源组应只引用虚拟设备和虚拟人员。缺少字段时表示真实资源。

虚拟资源用于表达本业务单位无法直接排产的转工、外协或无限能力占位环节。虚拟工序仍有有限的 `unit_duration_minutes`，但不发生设备和人员冲突，不检查日历、能力、容量、映射和资质。

虚拟设备、人员会显示在订单甘特图和明细中，并使用“虚拟资源”标签；它们不计入设备、人员数量、资源冲突和利用率。

## 5. 人工锁和局部拖拽

真实工序可以锁定设备、人员和时间。虚拟工序只允许锁定开始、结束时间，不提供设备、人员选择。

局部微调中：

- 真实工序可以调整当前资源组内设备、人员和时间；
- 虚拟工序只调整预计转工窗口；
- 虚拟工序不会与相同虚拟设备上的其他任务产生冲突；
- 调整上游工序时，下游虚拟工序及其后续工序会纳入影响范围。

## 6. 外部确认计划

外部业务单位确认虚拟工序时间后，可以在工序中维护 `external_plan`。静态、动态和局部三种模式都会将其作为固定任务保留，并以外部确认完成时间驱动后续工序。

外部确认计划和人工时间锁不能同时存在。外部确认时间的优先级高于算法预计时间。

`external_plan.source_business_type` 必须与虚拟资源组 `resource_group_type` 一致。

## 7. 三类业务日历

业务主数据必须明确维护三份日历：

| `schedule_type` | 用途 |
| --- | --- |
| `machining` | 机加白班、加班和停工规则 |
| `heat_treatment` | 热表连续生产窗口；热表不可中断工序必须完整落入一个连续窗口 |
| `assembly` | 装配班次和例外日历 |

完整业务快照使用 `machine_calendars` 数组传递三份日历。后端不再根据日历编号或名称猜测工艺类型，也不会启动时复制旧日历。新建算法任务时，业务系统按任务工艺选择其中一份，作为算法请求的单个 `machine_calendar`。

## 8. 管理建议

1. 正式部署先设置正确业务单元，再录入订单；
2. 跨业务转工工序必须绑定虚拟资源组；
3. `unit_duration_minutes` 应维护完整转工预计周期；
4. 不要为虚拟热表工序开启合批；
5. 外部确认计划返回后及时写回，并触发动态滚动排程；
6. 调试部署只用于联调和数据验证，不建议作为正式生产配置。
7. 上线前确认三份日历都显式维护正确的 `schedule_type`。
